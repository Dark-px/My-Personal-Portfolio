# SSR / SSG چیست؟ (راهنمای ساده)

## سوال

SSR/SSG یعنی چی و چه فرقی دارن؟

## جواب کوتاه

- **SSR** = رندر صفحه در سرور، در لحظه هر درخواست.
- **SSG** = تولید صفحه از قبل، در زمان build.

هر دو نسبت به CSR معمولاً برای SEO بهتر هستند.

---

## 1) CSR چیست؟ (وضعیت فعلی پروژه)

در مدل فعلی پروژه شما (Vite + React)، بیشتر رندر در مرورگر کاربر انجام می‌شود:

- ابتدا HTML اولیه سبک لود می‌شود.
- سپس جاوااسکریپت اجرا می‌شود.
- بعد محتوا (مثل پست‌های Medium) نمایش داده می‌شود.

این مدل به آن **CSR (Client-Side Rendering)** می‌گویند.

### مزایا

- پیاده‌سازی ساده
- UX سریع بعد از لود اولیه
- مناسب برای پنل‌ها/اپلیکیشن‌های تعاملی

### چالش برای SEO

- موتور جستجو باید JS را اجرا کند تا محتوا را ببیند.
- برای محتواهای بلاگی، نسبت به SSR/SSG ضعیف‌تر است.

---

## 2) SSR چیست؟

**SSR = Server-Side Rendering**

در SSR:

- کاربر درخواست صفحه می‌دهد.
- سرور همان لحظه HTML کامل تولید می‌کند.
- کاربر HTML آماده دریافت می‌کند.

### نتیجه برای SEO

- ربات موتور جستجو HTML آماده می‌بیند.
- ایندکس و درک محتوا بهتر و پایدارتر می‌شود.

### نکته

SSR برای صفحات با داده‌های دائماً در حال تغییر مفید است.

---

## 3) SSG چیست؟

**SSG = Static Site Generation**

در SSG:

- در زمان build، HTML صفحات از قبل ساخته می‌شود.
- در زمان درخواست، فایل آماده سریع تحویل داده می‌شود.

### نتیجه برای SEO

- بسیار خوب (مثل SSR)
- عملکرد عالی (Performance بالا)

### نکته

SSG برای محتواهایی که هر لحظه تغییر نمی‌کنند عالی است (مثل بلاگ، پورتفولیو، داکیومنت).

---

## 4) تفاوت SSR و SSG

- **زمان تولید HTML**
  - SSR: هنگام هر request
  - SSG: هنگام build

- **سرعت پاسخ**
  - SSR: خوب
  - SSG: معمولاً سریع‌تر

- **به‌روز بودن محتوا**
  - SSR: لحظه‌ای
  - SSG: وابسته به build (یا ISR/revalidate در بعضی فریم‌ورک‌ها)

---

## 5) برای پروژه شما کدام بهتر است؟

با توجه به هدف شما (Blog/Devlog + SEO بهتر):

- اگر می‌خواهید خیلی ساده و سریع مهاجرت کنید: **SSR**
- اگر محتوا بیشتر بلاگی و پایدار است: **SSG** گزینه بسیار خوبی است.

### پیشنهاد عملی

- اگر مهاجرت به Next.js/Nuxt-style stack مدنظر دارید:
  - صفحه `/blog` و صفحات مقاله را SSR/SSG کنید.
  - بقیه بخش‌های پورتفولیو می‌تواند استاتیک بماند.

---

## 6) تاثیر روی Medium Integration

اگر داده‌ها از Medium می‌آیند:

- در CSR: بعد از لود JS دیده می‌شوند.
- در SSR/SSG: می‌توانید پست‌ها را سرور-ساید بگیرید و HTML آماده تحویل دهید.

این یعنی:

- SEO بهتر
- اشتراک‌گذاری (OG) قابل‌اعتمادتر
- تجربه پایدارتر برای crawlerها

---

## 7) جمع‌بندی نهایی

- **CSR**: ساده و خوب برای UX، اما برای سئوی محتوایی ایده‌آل نیست.
- **SSR/SSG**: بهتر برای SEO، مخصوصاً برای صفحات Blog/Devlog.
- برای پروژه شما، ارتقای صفحه‌های بلاگ به SSR/SSG منطقی‌ترین قدم بعدی برای SEO است.
